# AI駆動でWebGPUプログラミング

ビルド不要のバニラJS/HTMLで作ったWebGPUシェーダープレイグラウンドです。`index.html`を開くとシェーダー一覧のホームページが表示され、各リンクから個別のシェーダーページに遷移します。

**デモ:** https://shugo15.github.io/webgpu-with-ai/

## 構成

- `index.html` — シェーダー一覧のホームページ
- `shaders/rotating-hue-gradient.html` — 画面中心を軸に回転しながら色相が変化するグラデーション
- `shaders/cornell-box-pathtracing.html` — SDFレイマーチング + パストレーシングによるCornell Box
- `shaders/cornell-box-specular.html` — Cornell Boxに完全鏡面反射・スペキュラ(グロッシー反射)の球を追加したもの
- `shaders/analytic-sky.html` — レイマーチングもモンテカルロ積分も使わない、解析的な高速近似による空(大気散乱)
- `shaders/ocean-gerstner.html` — ガーストナー波(トロコイド変位)の海面。実際の波の高さフィールドとレイの交差判定、フレネル反射(空の映り込み+水自体の透過色)で構成。`sky-precise.html`で焼き込んだ空のPNGを環境マップとしてインポートできる
- `shaders/ocean-mirror.html` — ガーストナー波の海面(roughness=0の完全な鏡面、`ocean-gerstner.html`と違い表面散乱・白波は無い)。空は`sky-precise.html`で焼き込んだequirectangular PNGのインポートが唯一のソース(解析的近似へのフォールバックなし)。波の法線を使った鏡面反射とフレネルに加え、海中の単散乱を吸収・散乱係数から閉形式で積分して水の色を物理的に計算する(色のハードコードなし)。海の深さをスライダーで指定でき、指定した深さの平らな海底(プロシージャルな砂地の模様)が浅瀬では波で歪みながら透けて見える

各シェーダーページは1つの`.html`ファイルに完結しており（WGSLはJS内の文字列として埋め込み）、サーバー不要でファイルを直接ブラウザで開けば動作します。

## Cornell Box レイマーチングの概要

`shaders/cornell-box-pathtracing.html`は、SDF（符号付き距離関数）によるレイマーチングと、簡易的なパストレーシングを組み合わせてCornell Boxを描画しています。

### SDF（符号付き距離関数）とレイマーチング

`map(p)`関数が、空間上の任意の点`p`から「最も近いオブジェクトの表面までの距離」を返すSDFです。壁や天井・床は平面までの垂直距離、内部の箱は`sdBox()`という定番の公式で距離を計算し、`opU()`（複数の距離のうち最小のものを選ぶ処理）で全オブジェクトを1つの距離関数にまとめています。

`raymarch()`はこのSDFを使ってレイを飛ばす手法（スフィアトレーシング）です。レイの現在位置から`map(p)`で最寄りの表面までの距離を求め、その距離ぶんだけレイを前進させる、を繰り返します。距離がほぼ0になったら表面にヒットしたと判定します。ポリゴンとの解析的な交差判定を解かなくても、SDFさえ定義すれば任意の形状に対して近似的にレイと表面の交点が求まるのがこの手法の特徴です。

### パストレーシングとコサイン重み付き半球サンプリング

拡散反射（艶消しのマットな表面）は、光が当たった点から法線を軸にした半球状のあらゆる方向へランダムに反射します。パストレーシングでは、この挙動を乱数でランダムな方向を1つ抽選してレイを継続することでシミュレートします（`cosineSampleHemisphere()`）。「コサイン重み付き」とは、法線に近い方向ほど選ばれる確率を高くするサンプリング方法です。

レンダリング方程式をモンテカルロ法で推定すると次の形になります。

```
L_o ≈ (1/N) Σ [ f_r · L_i · cosθ / pdf(ω) ]
```

ここで`cosθ`は法線とサンプリング方向の内積です。Lambertian BRDF `f_r = albedo / π`と、コサイン重み付きサンプリングのPDF `pdf(ω) = cosθ / π`を代入すると、

```
f_r · cosθ / pdf(ω) = (albedo/π) · cosθ / (cosθ/π) = albedo
```

`cosθ`が分子・分母で相殺されます。そのため実装上は`throughput *= albedo`という単純な掛け算だけで済んでおり、`dot(法線, サンプリング方向)`を別途明示的に計算する必要がありません。内積（cosθ）が消えたわけではなく、「法線方向に近いほど選ばれやすい」というサンプリングの偏りとしてコードに暗黙的に埋め込まれている、という点がポイントです（もし全方向一様サンプリングにした場合は`pdf = 1/(2π)`固定になるため、`albedo * dot(法線, 方向) * 2.0`のように明示的にcosθを掛ける必要があります）。

### 時間方向の積分（累積平均）によるノイズ収束

カメラは時間で動かさず固定したまま、フレームごとに新しいノイズ入りのパストレーシング結果を、前フレームまでの累積平均に混ぜ込んでいきます（ping-pongさせた2枚の`rgba16float`テクスチャで実装）。

```
newAvg = prevAvg + (currentSample - prevAvg) / n   // n = 累積フレーム数
```

これは総和を保持せずに逐次的に平均を更新する定番の手法で、フレームを重ねるほどモンテカルロサンプリングによるノイズが打ち消されて画像が滑らかに収束していきます。画面右上のスライダーで「反射回数上限」「レイ数/pixel」を変更すると、蓄積フレーム数が0にリセットされ、新しい設定で収束し直します。

## ガーストナー波の海の概要

`shaders/ocean-gerstner.html`は、ガーストナー波(トロコイド変位)による海面を、透過なしの鏡面反射(反射+フレネルによる透過色のブレンド)で描画しています。

### 実際の高さフィールドとの交差判定

`gerstnerSample()`が波の法線"と"実際の高さ(垂直変位)を1回のループでまとめて計算します。`intersectWater()`はこの高さフィールドに対して、レイに沿って区間探索(近距離は細かく、遠距離は粗いステップ)を行い、符号が変わった区間を二分探索で精密化することで交差点を求めます。反射(1回目も含む)は「反射回数」スライダーの上限まで繰り返し、それでも下向きのままなら水面が水面自身を映しているとみなしたフォールバック色(または強調表示用の色)を使います。

### 透過色(水そのものの色)の決め方

フレネルで反射と混ぜる透過色は、次の2つの積として決めています。

```
transmissionColor = waterColor(水自体の吸収・散乱による平衡色) × sunVisibility(太陽の可視度)
```

**水自体の色**は、深い(底が見えない)海を想定すると、吸収と散乱が釣り合うところで深さに依存しない一定の色に収束します。水は波長によって吸収される速さが大きく異なり(赤は数mでほぼ吸収、青は数十m以上届く)、この差が「深い海は青い」という現象の物理的な理由です。デフォルト値は、純水の吸収スペクトルに関する代表的な文献であるPope & Fry (1997, *Applied Optics* 36, 8710–8723)で報告されている吸収係数のおおよその値(赤 650nm付近で約0.32 m⁻¹、緑 550nm付近で約0.06 m⁻¹、青 465nm付近で約0.018 m⁻¹)をもとに、各チャンネルの吸収係数の逆数の比(≒その波長がどれだけ深くまで届き、散乱されて戻ってきやすいか)を色の比率として使っています。

**太陽の可視度**は空のモデルと同じ`sunVisibilityOf(sunDir)`を再利用し、太陽が沈んでいる夜間は水中に光が届かないため透過色がほぼ黒に近づくようにしています。色相自体(青み)は水そのものの吸収特性が支配的なので、太陽光が夕焼けで赤くなってもそれに引きずられて赤くはならず、明るさだけが太陽の状態に連動します。

### 焼き込んだ高精度の空を環境マップとして反射に使う

既定では反射に映る空は解析的近似(`analytic-sky.html`と同じモデル)ですが、「焼き込み空(PNG)をインポート」ボタンから`sky-precise.html`でエクスポートしたequirectangular PNGを読み込むと、多重散乱パストレーシングによる高精度の空を環境マップとして反射・空の描画に使えます(`file://`で開いてもCORSに引っかからないよう、`fetch`ではなくファイル選択ダイアログ+`createImageBitmap`で読み込む方式)。

エクスポートされたPNGはReinhardトーンマップ+ガンマ補正済みのLDR(8bit)なので、サンプリング時にガンマ2.2とReinhardの逆変換`m/(1-m)`でおおよその線形HDR放射輝度に戻してから反射計算に使います。LDRで1.0に飽和した画素(太陽近傍の明るい空など)は逆変換が発散するため上限をクランプしています。方向→UVの変換(`dirToEquirectUV`)と太陽の軌道式は`sky-precise.html`と同じものを複製しているため、焼き込み時の時刻を使えば、環境マップ内の太陽と、水面のグリッター・透過色の明るさ(これらは時刻から計算する)が同じ方向で整合します。この時刻はPNGに埋め込まれたメタデータからインポート時に自動設定されます(後述)。なお現在の`sky-precise.html`は太陽円盤をPNGに焼き込まない方式(後述)のため、インポートした空を使うときも解析空と同じ太陽円盤項を`skyRadiance()`で加算しています(輝度は解析空と同スケールに抑え、粗い水面の反射レイが円盤を掠めたときのファイアフライを増やさないようにしています)。

## ガーストナー波の鏡面の海(`shaders/ocean-mirror.html`)の概要

`shaders/ocean-gerstner.html`との違いは、当初「波を一切持たない完全に平らな水面(y=0の無限平面)」だったが、後述の通りガーストナー波を導入したことで「roughness(表面の粗さによる散乱)を持たない」ことに一本化された。水面上のどの点でも(波で傾いていても)完全な鏡面として振る舞うため、乱数によるレイの散らばりが発生せず、1ピクセルにつき視線は常に1本で済む。空は解析的近似へのフォールバックを持たず、`sky-precise.html`で焼き込んだequirectangular PNGのインポートが唯一のソースになっている。水面下の海中については、水の色をハードコードせず、海水の光学係数から単散乱を閉形式で積分して色を計算する(後述)。

### (当初)位置に依存しない反射方向 → ガーストナー波の追加で交点計算が必要に

当初は無限に平らな水平面(法線が常に`(0, 1, 0)`で一定)として実装しており、視線ベクトルの反射方向は「どこで反射したか」に依存せず、視線のy成分の符号を反転するだけで求まっていた(`reflDir = vec3(rayDir.x, -rayDir.y, rayDir.z)`)。そのためレイと水面の交差点そのものを計算する必要が無く、`ocean-gerstner.html`のような区間探索+二分探索のレイマーチングが丸ごと不要だった。

その後「屈折がわかりやすいように波を入れたい」という要望を受け、`ocean-gerstner.html`と同じガーストナー波(トロコイド変位)の高さフィールドを導入した。波を入れると法線が位置ごとに変わるため、この位置非依存の近似はもう使えず、`ocean-gerstner.html`と同じレイマーチング+二分探索(`intersectWater()`)でレイと波面の実際の交点を求める構成に戻した。見た目の忠実度を優先する判断で、「バンプマップ的に交点は平面のまま法線だけ波から取る」という速度優先の代替案(検討はしたが不採用)よりも精度を選んだ形になる。

ただし乱数によるノイズは相変わらず発生しない(表面粗さによる散乱が無いため、`intersectWater()`のレイマーチングも二分探索も決定的な計算)。そのため時間積分による蓄積(ping-pongバッファ)は不要なままで、フレームごとに交点を求め直すだけのシンプルな1パス構成を維持している。波のアニメーションも、`ocean-gerstner.html`が持つ「粗さによるノイズが波を動かすと蓄積し直しになる」という制約が無いため、一時停止ボタンを設けず常時動かし続けている。

### ガーストナー波の導入: 屈折を目に見える形にする

平面のままだと、屈折はすべて`waterRadiance()`内部の光学計算に閉じており、画面上で「屈折している」と分かる見た目の変化が無かった(海底の模様も真上から見れば真下にそのまま透けて見えるだけ)。ユーザーから「屈折がわかりやすいように波を入れたい」と要望を受け、まず「波は結局ガーストナー波になるのか」を検討した。

波の**形**は`ocean-gerstner.html`のガーストナー波(同じ波長・振幅・向きの8波の重ね合わせ)をそのまま流用するのが自然と判断した。ゼロから別の波モデルを作る理由が無く、既にこのリポジトリでチューニング済みだったため。一方**計算方法**については2つの選択肢を検討した:

1. **バンプマップ近似(速度優先):** レイと水面の交点は従来通り平面`y=0`との代数的な交点のまま求め、その位置でガーストナー波の法線だけを評価して反射・屈折に使う。レイマーチング不要で1パス構成を保てるが、真の交点(実際に波打った高さでの交点)とはズレる近似
2. **レイマーチングによる真の交点(精度優先):** `ocean-gerstner.html`と同じ区間探索+二分探索(`intersectWater()`)でレイと波面の実際の交点を厳密に求める

最初は不要な複雑さを避けるためバンプマップ近似を提案したが、ユーザーから「速度より見た目の忠実度を優先したい」という方針が示されたため、2番目の真の交点を求める方式を採用した。

実装は`gerstnerSample()`(法線・高さのみ。このシェーダーには白波が無いのでラプラシアン・大波の高さは計算しない)と`intersectWater()`を`ocean-gerstner.html`から移植し、`waterRadiance()`の引数に局所法線`normal`を追加して、これまで固定`up = vec3(0,1,0)`だった箇所(視線・太陽光の屈折、フレネル)をすべて`normal`に一般化した。海底の深さは「指定した深さ + 交点の高さ`hitPos.y`」で局所的に補正する(波の頂上ではその分深く、谷ではその分浅くなる。海底自体はワールド空間で波と無関係に平らなまま)。太陽光の水面透過(`eUnderwater`のフレネル項)も、それまで`sunDir.y`(ワールド鉛直角)を使っていたのを`dot(sunDir, normal)`(局所法線とのなす角)に修正し、傾いた波面での入射角を正しく反映するようにした。

結果として、真上から見ても海底の模様が波にあわせて歪んで揺らめき、深い場所では空の反射(直線であるはずの帯)が波でさざ波状に崩れて見えるようになった。ヘッドレスでの検証では、swiftshaderソフトウェアレンダリング・260×260でも約60fps(vsync上限)を維持でき、波の追加によるレイマーチングの追加コストは実用上問題にならないことを確認した。

波を導入した直後は`intersectWater()`の探索範囲(これを超えると水面が見つからず空とみなす)を固定値100mにしていたが、平面だった頃と違って波のせいで遠方ほど交点が見つかりにくくなる(浅い視線角度では特に)ため、`shaders/ocean-gerstner.html`の「描画距離」スライダーと同じ形(20〜500m)で調整できるようにした。20mでは水平線直下の帯がintersectWater()の探索失敗によりテスト画像の地面色にフォールバックし(輝度が完全に一定の20.0になる)、500mではそれが消えて波の起伏が続くことをピクセル値で確認した。

### 描画距離を伸ばすと手前の精度まで落ちないか(遠距離側のステップ幅の上限)

描画距離スライダーを追加した直後、ユーザーから「負荷が上がっていいので描画距離を伸ばしても手前の精度を落とさないでください(ステップ幅が広くなっている?)」という懸念が出た。`intersectWater()`のコードを確認すると、近距離側(`nearDist=5.0`・`nearSteps=28`)は`maxDist`と無関係な固定値なので、理屈の上では手前のステップ幅は描画距離を伸ばしても変わらないはずだった。ただし推測で済ませず、波のアニメーションを`performance.now()`を固定値に差し替えて静止させた上で、描画距離20mと500mでレンダリングした画像の手前(画面下25%)を比較したところ、平均絶対差は0.15/255(丸め誤差レベル)で、画面全体の平均絶対差3.28よりずっと小さいことを確認した。差はほぼ全て遠方に集中しており、手前の精度は実際に保たれていた。

一方で、遠距離側のステップ数は`ocean-gerstner.html`から移植した固定24ステップのままだったため、`farStep = (maxDist - 5) / 24`は描画距離に比例して粗くなる(500mでは約20.6m/ステップとなり、最も波長の短い「大きなうねり」4.5mより大きい)という別の問題が実際にあった。これは「手前」ではなく「遠方」の精度劣化だが、ユーザーの「精度を落とさないでください」「負荷が上がっていいので」という方針に沿って、遠距離側のステップ数も動的に`ceil(farDist / TARGET_FAR_STEP)`(下限8本、`TARGET_FAR_STEP=1.5m`)で確保するよう修正し、描画距離をどれだけ伸ばしても遠方のステップ幅が一定以下に収まるようにした。

### 波の形状をリファレンス写真に近づける(固定8波→統計的な広帯域スペクトル)

ユーザーから外洋のチョップの多い波の写真を提示され、「波の形状に不満がある。リファレンス画像のような形状を目指したい」という指摘を受けた。手で選んだ大きなうねり3つ+さざ波5つの計8波では、リファレンスにある「あらゆるスケールの波が無数に重なった不規則な質感」「風向きに沿った筋状のチョップ」を再現できていなかった。範囲を「波の形状のみ」(白波・泡は別範囲)に絞ってユーザーに確認した上で、固定配列で波を列挙する方式から、ハッシュ関数による疑似乱数で手続き的にNUM_WAVES本を生成する方式に置き換えた:

- **波長**: `WAVE_MAX_LENGTH`(6m)から`WAVE_MIN_LENGTH`(4cm)まで対数的(等比数列)に分布させ、本ごとに±15%のランダムな揺らぎを加えて周期的なパターンが出ないようにする
- **振幅**: 傾き(振幅/波長の比、`WAVE_STEEPNESS`)をほぼ一定にする。これは「大小どのスケールの波もだいたい同じくらい険しく見える」という、海洋工学のPhillipsスペクトルが表す自己相似性を単純化して模したもの
- **向き**: 支配的な風向き(海底の砂紋と同じ向きに揃えた`WIND_DIR`)を中心に、波長が短いほど広がり角を大きくする(現実の風波も、支配的なうねりは風向きに揃うが細かいさざ波ほど向きが乱れる)
- **位相**: 波ごとにランダムなオフセットを持たせ、原点で全波が同位相になる不自然さを防ぐ

**ヘッドレス検証中に踏んだ罠: NUM_WAVES=17以上で確実にハングする。** 当初NUM_WAVES=20で実装しヘッドレス検証したところ、`toDataURL()`の呼び出しがCPU使用率ほぼ0のまま何十秒経っても返ってこないという現象に遭遇した。描画距離やビューポートを小さくしても再現したため負荷の問題ではないと判断し、NUM_WAVES自体を2分探索したところ、**16は必ず成功し17は必ずハングする**という明確な境界を確認した(swiftshaderソフトウェアレンダリングのシェーダーコンパイラかドライバの何らかの上限に当たっていると見られるが、根本原因はWGSL側からは特定できない)。実機の本物のGPUでは発生しない可能性もあるが、検証できない環境まで含めて安全側に倒し、実証済みの16に留めている。

**教訓:** 「反応が返ってこない」ときに時間切れで諦めて次の手を打つのではなく、まず「本当にハングしているのか、単に遅いだけか」を切り分けることが重要だった。実際、後続の検証(NUM_WAVES=16・実際のデフォルト設定)で一度「50秒待って応答なし」と判断してプロセスを強制終了させたところ、裏では既に正常終了していたことが判明した(タイムアウトを100秒に伸ばして再実行したら普通に完了した)。ヘッドレス環境(特にソフトウェアレンダリング)はシェーダーのコンパイルや初回描画が実機よりかなり遅いことがあるため、「反応が無い=壊れている」と即断せず、まず待機時間を伸ばして再現するかを確認するべきだった。

### カメラを高くしたら近距離域の設計前提が崩れて激重になった

ユーザーから「カメラの位置を高くして」と要望され、固定カメラの高さ(`eye.y`)を2.5mから引き上げた。ところが検証すると、レンダリングが数分単位まで極端に遅くなる現象が起きた。

原因は`intersectWater()`の近距離/遠距離の区間分けにあった。近距離域`nearDist`は「カメラの高さが2.5m程度」であることを暗黙の前提に固定値5.0mとしていたが、カメラを高くすると、同じ画面位置(同じ視線角度)のレイがy=0に届くまでのスラント距離は**カメラの高さにほぼ比例して伸びる**。そのため多くのレイが、精密だが範囲の狭い近距離域を素通りして、範囲は広いが1ステップが粗く本数も多い遠距離域を律儀に(場合によっては描画距離の上限近くまで)探索することになり、平均コストが跳ね上がっていた。対策として近距離域`nearDist`をカメラの高さに比例させ(`max(ro.y * 2.0, 5.0)`)、多くのレイが再び近距離域だけで収まるようにした(本数`nearSteps`は据え置き、範囲が広がった分だけ1ステップの幅がわずかに粗くなるトレードオフは許容した)。

あわせて、水平線に近い浅い角度のレイは「描画距離内ではどの波面にも数学的に絶対届かない」ケースが増える(カメラが高いほど顕著)。このケースは以前は遠距離側を律儀に全ステップ探索してから「見つからなかった」と判定していたため最も重いパターンになっていた。レイの終点(`t = maxDist`)での高さが、全波の振幅の合計より十分大きい安全マージン(`WAVE_HEIGHT_MARGIN = 3.0`)を超えている場合はマーチングを一切せず即座に打ち切るようにし、この最悪ケースを避けた。

**教訓:** 「カメラ位置」のような一見無害なパラメータでも、それを暗黙の前提にしたチューニング済みの定数(このケースではnearDist)が他所に埋め込まれていると、変更が思わぬ場所で性能崩壊を引き起こす。固定値を使う場合は、それが何の値に対して妥当なのか(このケースでは「カメラの高さ2.5m程度」)をコメントに残しておくべきだった。

### フォールバック箇所の強調表示

`intersectWater()`が描画距離内に波面との交点を見つけられなかった場合、そのレイは(反射・屈折・海底ぜんぶ計算せずに)単純に空とみなしてサンプリングする。このフォールバックは描画距離が足りていない水平線際で起きやすく、実際に描画距離20mの検証で水平線直下に黒い帯として現れた(前述)。しかしこの帯は空の色によっては目立たず(たとえば空自体も暗ければ気づきにくい)、どこがフォールバックなのか一見して分からないという問題があった。

`shaders/ocean-gerstner.html`の「失敗箇所を強調表示」(反射回数を使い切った箇所を単色で塗る)と同じUIパターンで、`intersectWater()`が交点を見つけられなかった箇所を任意の色(デフォルト マゼンタ)で塗りつぶすチェックボックス+カラーピッカーを追加した。これにより、描画距離が足りているかどうかを一目で判断できる(強調表示を有効にして帯が全く見えなければ、その描画距離で十分ということになる)。

### 反射失敗箇所(急な波の傾きで反射方向が水面へ戻ってしまう)の発見と強調表示

この強調表示を実装した後、ユーザーから「水面の反射が失敗した部分は?」と問われて確認したところ、`intersectWater()`の探索失敗とは別種の、より根深い問題を発見した。反射方向`reflDir = reflect(rayDir, normal)`の計算結果を一度も検証せず、そのまま`sampleSky(reflDir)`していたのである。波の傾きが急な箇所(広帯域スペクトル化で増えた高周波の細かい波の頂上・谷に多い)では、`reflDir`が上(空)ではなく下(水面へ戻る方向)を向くことがある。本来なら`shaders/ocean-gerstner.html`のように反射方向をさらに追跡する(多重反射)べきだが、このシェーダーは1回の反射しか計算しないため、この場合`sampleSky(reflDir)`は空テクスチャの下半分(`sky-precise.html`では地面の暗い色が焼き込まれている領域)を誤ってサンプリングしてしまっていた。

強調表示を有効にして確認すると、水面のあちこち(特に細かい波の頂上・谷)に無数のマゼンタの筋が現れ、この問題が広く発生していたことが分かった。修正として、`reflDir.y < 0`を検出し、その場合は`shaders/ocean-gerstner.html`の同種のフォールバックと同じ「水面が水面自身を映す、それらしい暗い水色」(`vec3(0.02, 0.05, 0.08)`)を使うようにし、太陽ハイライトの計算(`sunDisc`)もこのケースではスキップするようにした。強調表示を有効にした場合は、この箇所を`intersectWater()`の探索失敗と同じ強調表示色で塗りつぶす(同じチェックボックス+カラーピッカーを流用し、UIを増やしていない)。

**教訓:** ある種の「フォールバックが必要な箇所」を可視化する仕組みを作ったこと自体が、「他にも検証していない前提があるのでは」というユーザーの問いかけを呼び、実際に別の(より発生頻度の高い)未検証の前提を発見する結果になった。ベクトル演算の結果(この場合`reflect()`の戻り値)は、その後の処理が暗黙に仮定している範囲(この場合「上向き」)に本当に収まっているか、実装時に一度は明示的に確認すべきだった。

### フォールバック色がフレネル反射率の高い遠方で目立つと判明し、多重反射で解消

上記のフォールバック(反射方向が下向きになるケース)を固定の暗い水色で塗る対症療法を入れた直後、ユーザーから「黒くなっている部分はどうなっているのか、水深を浅くしても黒く出るということはフォールバックしていると考えられる」と指摘された。実際に検証すると、単純に「フォールバック=黒」ではなく、**フレネル反射率が高い(視線が水面に対して浅い角度=遠方)領域でだけ**顕著に暗くなっていることが分かった。フォールバック色は`sky * fresnel`という項に使われるため、フレネル反射率が高い遠方ではこの項の寄与が支配的になる一方、`(1 - fresnel)`で重み付けされる海中散乱(明るい海底を含む)の寄与は相対的に小さくなる。近傍(フレネル反射率が低い)ではフォールバック該当ピクセルと非該当ピクセルの平均輝度はほぼ同じ(135.0 vs 132.3)だったのに対し、遠方では大きく乖離した(32.3 vs 118.0、最小輝度18.0)ことをピクセル値で確認した。この乖離は「水深を変えても黒いまま」というユーザーの観察と一致する(フォールバック色自体が`seaDepth`に依存しないため)。

固定色によるフォールバックはこのように「目立たない場所では問題ないが、目立つ場所では悪目立ちする」という対症療法の限界を露呈したため、根本対応として`shaders/ocean-gerstner.html`と同じ多重反射を実装した。反射方向が下向きのままなら、その方向へ`intersectWater()`をもう一度呼んで次の波面との交点を求め、そこで反射方向を計算し直すことを`MAX_REFLECTION_BOUNCES`(3)回まで繰り返す。roughness=0の鏡面なので乱数は不要で、`intersectWater()`を複数回呼ぶだけで決定的に追跡できる。強調表示で計測したところ、多重反射の追加により未解決のフォールバック(強調表示される箇所)は3.03%から0.00%まで低減した(浅瀬・描画距離100mの検証)。

**教訓:** 症状(黒い部分)を指摘されたときに「フォールバックだから」で終わらせず、実際にどの条件でどれだけ目立つかを測定したことで、「対症療法(固定フォールバック色)では条件によって悪目立ちする」という、より根本的な設計上の限界が見えた。これが結果的に、単純な検出+強調表示に留めていた前項の対応から、根本原因を解消する多重反射の実装へ踏み込む決め手になった。

### 太陽の輝き: 焼き込み画像の反射だけでは不安定なため、明示的なハイライト項を追加

理屈上は完全な鏡面なので、焼き込み画像内の太陽円盤が反射方向にそのまま映り込むだけで太陽の輝きが表現されるはずだった。しかし実際に試すと、equirectテクスチャ上での太陽の角サイズはわずか数ピクセルしかなく、フレームバッファ解像度・4サンプルのAA・bilinear補間だけでは、カメラの向きによってハイライトが消えたりチラついたりして安定して見えないことが分かった。

対策として、テクスチャの中身に頼らずsunDir(スライダーの時刻)から直接ハイライトの明るさを計算する項を追加した。その後`sky-precise.html`側も太陽円盤をPNGに焼き込まない方式(後述)へ変更したため、現在はテクスチャに太陽が含まれておらず、この解析円盤が直視(空)・反射(水面)両方の太陽の唯一のソースになっている。

**最初の実装ミス:** `ocean-gerstner.html`の`sunGlitter()`と同じBlinn-Phong方式(視線ベクトルと太陽方向のハーフベクトル`H=normalize(view+sun)`と法線の内積のべき乗)をそのまま流用したところ、ユーザーから「完全に平らな水面なら太陽の形は歪まずそのまま映るはずでは?」という指摘を受けた。実際、Blinn-Phong方式は水面全体に無数の異なる傾きの微小面がある「粗い」水面を前提にした近似であり、視線が水平線に近い浅い角度になるほどハーフベクトルと法線のなす角が、真の反射角(反射方向と太陽方向のなす角)からずれていく性質がある。roughnessゼロの完全な平面にこのズレをそのまま持ち込むと、水平線に近づくほどハイライトが手前に間延びした涙形(いわゆる写真の"sun glitter path")になってしまい、平らな鏡面では本来歪まず円形に映るはずの太陽の形と矛盾していた。この涙形は`ocean-gerstner.html`の波のある水面でこそ物理的に正しい近似であり、roughnessが存在しない平面シェーダーに移植したこと自体が誤りだった。

**2回目の修正:** 反射方向`reflDir`と太陽方向`sunDir`の「なす角」だけに依存する円盤の式(`shaders/sky-precise.html`の太陽円盤`smoothstep(cos(0.02), cos(0.009), dot(rayDir, sunDir))`と同じ考え方)に置き換えた。この式は視線角度に関係なく常に真円のハイライトになる。実際に反射側ハイライトの外接矩形の縦横比を測定したところ、鋭さの設定によらずほぼ1.0(0.99〜1.11程度)になることを確認した。この時点では円盤の角半径を対数的に制御する「ハイライトの鋭さ」スライダーを設け、水面の物理的な粗さではなく、写真でも実際に観測される(カメラ/眼のレンズによるにじみに相当する)見た目の輝きの広がりを表す演出的なパラメータとして導入していた。

**3回目の修正(鋭さスライダーの撤廃):** その後ユーザーから「この鋭さの数値、そもそもglossy(粗さ)の意味なのか?」と問われ、水面が常にroughness=0の完全な平面鏡であることを踏まえて検討した結果、「鋭さ」は不要と判断した。水面が常に完全な鏡面である以上、反射に映る太陽の唯一物理的に正しい大きさは、直視したときと同じ実際の角半径(`SUN_ANGULAR_RADIUS`、0.02ラジアン)であり、それ以外の値は「反射だけ意図的に滲ませる」という、この水面の前提(roughness=0)と矛盾する演出でしかなかった。半径を`SUN_ANGULAR_RADIUS`固定に戻し、スライダーごと削除した。あわせて、このハイライト項がフレネル反射率で重み付けされていなかった(空の反射`sky * fresnel`は視線角度に応じて2%〜100%に変化するのに、太陽のハイライトだけ角度に関係なく一定の強さで加算されていた)ことにも気づき、`disc * fresnel * highlightIntensity`とフレネルを掛けるよう修正した。「ハイライトの強さ」だけが、反射側の太陽を直視側と独立してオフ/減衰できる演出用のノブとして残っている。

**教訓:** 別のシェーダー(粗い水面)で正しく機能しているコードでも、前提条件(roughnessの有無)が違う場所にそのまま持ち込むと物理的に矛盾した結果になりうる。1回目の修正でユーザーの「平らならこうなるはず」という物理直感からの指摘が見た目のバグに気づく決め手になったのに続き、3回目の修正でも「この数値は何を表しているのか」という同種の問いかけが、一度は正しく動いていたが本来不要だった演出パラメータと、その裏に隠れていた別の物理的な見落とし(フレネル重み付け漏れ)の両方を洗い出す結果になった。

なお後述の通りこのシェーダーには後にガーストナー波を導入したが、「roughness=0の完全な鏡面」という前提自体は保たれている(波で局所法線は変わるが、各点は依然として粗さの無い鏡面)ため、この節の結論(直視・反射どちらもSUN_ANGULAR_RADIUS固定、フレネル重み付け必須)はそのまま成り立つ。

### 波の導入後もハイライトが「弱い」ままだった問題: Cox-Munk統計モデルへの置き換え

ガーストナー波(前述)を導入し、実際の海の写真(江ノ島の砂浜で撮影した、太陽が低く水面に幅広くきらめきが伸びる参考写真)と見比べたところ、ユーザーから「太陽光のハイライトが弱く見える」と指摘された。上記の`sunDisc()`(反射方向と太陽方向のなす角だけで決まる真円のハイライト)は、局所法線がたまたま厳密に太陽を向いた点だけを光らせる二値判定であり、`NUM_WAVES`本(swiftshaderの上限で16本)の重ね合わせでは局所法線の向きの分布が疎で、参考写真のような無数の細かいきらめきが連なる帯にはならなかった。

最初、カメラのレンズ特性(にじみ・ブルーム)による見た目の補正を提案したところ、ユーザーから「カメラのレンズ特性ではなく、水面側の物理の話をしたい」と明確に方向修正された。水面自体の物理的性質としてハイライトを強く・広くする根拠が必要だった。

そこで採用したのが**Cox & Munk (1954)の統計的太陽グリッターモデル**である。実際の海面には(理論上は連続スペクトルの)無数の波が重なっているが、レンダリングでは有限本の波しか重ね合わせられないため、個々の点の法線が正確に太陽を向くかを判定する方式は原理的に疎になる。Cox & Munkは実際の太陽グリッターの写真から、風で波立った海面の傾きの分布がほぼガウス分布に従うことを実測で示した。これは物理ベースレンダリングの**Beckmann分布**(マイクロファセットBRDFの法線分布関数)と数学的に同じものであり、個々の面の向きを1つずつ判定する代わりに、傾きの分散`sigma2`を使って「この点が統計的に太陽方向を向いている確率密度」を連続的に評価できる。

ユーザーから「その確率密度に太陽光の輝度をかけるということですか?トレースはせずに」と実装方針の確認を受け、その通りであることを説明した上で実装した。ポイントは以下の2点:

1. **`sigma2`は新しい自由パラメータではない**。呼び出し側では、交点の`WaveSample.unresolvedSlopeVar`(下記「波のLOD」参照。この画素がLODフェードで幾何形状から取り除いた、解像できていない細かいスケールの傾きの分散)をそのまま渡す。全波スペクトルの分散をまとめて使う案も検討したが、それだと**大スケールの波の傾き(=`reflect()`による幾何学的な鏡面反射に既に正しく反映されている分)まで統計モデルにも二重に足し込んでしまう**。解像できていない分だけを渡すことで二重計上を避けつつ、カメラに近い(幾何形状で十分解像されている)場所では`sigma2`が0に近づいて統計的ハイライトはほぼ幾何反射任せになり、遠い(LODで細部が消える)場所ほど`sigma2`が大きくなって統計的なきらめきが支配的になる、という距離に応じた自然な遷移になる
2. **レイトレースは一切行わない**。Cook-Torranceのマイクロファセット鏡面BRDF`f_r = D(H)F(V,H)G / (4 cosθ_V cosθ_L)`を太陽という直接光源に適用すると、レンダリング方程式`L_o = f_r × E_sun × cosθ_L`の`cosθ_L`と分母の`cosθ_L`が相殺され、`L_o = D(H) F(V,H) E_sun / (4 cosθ_V)`という、確率的サンプリングを伴わない解析的な直接評価の式になる(`sky-precise.html`のNext Event Estimationと同じ考え方)。幾何減衰項`G`(マイクロファセット同士の遮蔽)は既知の簡略化として省略している

`sunDisc()`自体は削除せず、「直視の太陽本体」と「多重反射で反射方向を厳密に追跡した場合のハイライト」で引き続き使う設計にしたが、後述(教訓その5)の通りこの時点の実装では反射側の呼び出しを実際には削除してしまっていた。統計モデルに置き換えるつもりだったのは、水面反射側の主要なハイライト(`sunGlitter()`)だけである。

検証は`synthetic_equirect.png`を使ったヘッドレスレンダリングで、置き換え前後を比較した。太陽高度が低い時刻(朝7〜8時相当)では、変更前はほぼ何も見えなかった水面のハイライトが、変更後は水平線際から手前に向かって伸びる涙形の明るいきらめきの帯(カメラに近いほど幾何反射で細く、水平線に近づくほどLODで統計的に太くなる)として現れることを確認した。太陽高度が上がるほど(9時相当)帯は狭く小さくなることも確認でき、これはCox-Munkモデルが予測する「太陽が高いほどグリッターパスは短く狭くなる」という性質と一致する。

**教訓その1:** 「弱く見える」という見た目の指摘に対して、最初にカメラ(観測系)側の効果を提案してしまったが、ユーザーは水面(被写体)側の物理的な性質の話をしていた。同じ「明るく見せる」という結果でも、原因をどちらに帰属させるかで実装のアプローチも物理的な正しさも全く変わる。

**続報1: 今度は「強すぎる」問題と、Cox-Munk自身の式による下限の導入**

統合直後にユーザーから今度は逆に「ハイライトが強すぎる」と指摘された。原因は`sunGlitter()`のBeckmann分布のピークが`1/(π・sigma2)`に比例することで、`sigma2`(=`unresolvedSlopeVar`)がカメラに近い、幾何形状でほぼ解像された場所ではほぼ0に近づき、ピークが際限なく尖ってしまっていた点にあった。発散防止のためだけに置いていた下限(`1e-5`)はあまりに小さく、実質的に無制限を許してしまっていた。

これを、意味のある物理量に置き換えた: 実際の海面はこのシェーダーが表現する最も細かい波(`WAVE_MIN_LENGTH`=4cm)よりさらに細かいミリメートル級の毛細波(capillary wave)を常に持ち、どんなに近くで見ても真の意味で完全な鏡面にはならない。Cox & Munk (1954)自身が示した風速依存式`sigma2 = 0.003 + 5.12×10⁻³×W`(`W`=風速m/s)は、無風(`W=0`)でも`sigma2=0.003`という非ゼロの値を与えており、これがまさに「常に存在する毛細波によるぼやけ」の下限を表している。この`0.003`を`CAPILLARY_SLOPE_VAR`として下限に採用したところ、水面近くの過飽和が収まり、水平線際に幅広い帯が広がりすぎていた見た目が、中心の明るい核から自然に減衰する、より写真に近いグリッターパスになった。

**教訓その2:** 発散防止のためだけに置いた「とりあえずの小さい値」の下限(`1e-5`)は、コンパイルは通り一見動いているように見えても、値そのものに物理的な意味がないため、結果の見た目(強すぎる/弱すぎる)を左右する重要なパラメータになっていることに気づきにくい。ここでも、参照している論文(Cox & Munk 1954)自体が同じ問題(sigma2=0のときの不定)に対する答えをとうに用意していた(風速0の極限値)ため、それをそのまま使うだけで済んだ。

**続報2: それでも「まだ強いし太い」— 実は表示空間へトーンマップ済みの色に線形HDRを無変換のまま加算していたバグ**

上記の下限を入れても、参考写真(江ノ島とは別の、うねりが穏やかで太陽が霞んだ写真)と見比べたユーザーから「まだまだ強いし太い」と重ねて指摘された。今度は`CLAUDE.md`が推奨する「WGSLと同じ計算をNode.jsに複製して決定的に検証する」方法で`sunGlitter()`の生値(トーンマップ前)をカメラからの距離ごとに数値で追跡したところ、水平線付近から手前10m前後まで、広い範囲でほぼ一様に1.0前後の値が続く「台地」になっていることが分かった(距離とともに`sigma2`は増えてD(H)のピークは下がるが、同時に視線が水面に対して浅くなって`cosθ_V`が小さくなり、式の分母`4cosθ_V`で増幅する分とちょうど打ち消し合っていた)。

さらにコードを見直すと、真の原因は物理モデルではなく実装のバグだった: `underwater`(海中散乱、線形HDR)は`tonemapGamma()`で表示空間(0〜1)に変換してから`sky * fresnel`(既に焼き込み済みの表示空間の値)に加算していたのに対し、`sunGlitter()`の寄与(`glitterRadiance`、これも線形HDR)は**トーンマップを一切せず、既に表示空間になった`waterColorResult`へ直接加算していた**。1.0前後の線形値が表示空間の色にそのまま乗ると、キャンバスの表示範囲(0〜1)を大きく超えて一律に真っ白へクリップされる。これが「強い」だけでなく「太い(広い範囲が同じ真っ白になる)」の直接の原因だった。修正は、`underwater`と`glitterRadiance`を線形のまま先に合算し、`tonemapGamma(underwater + glitterRadiance)`として1回だけトーンマップしてから、表示空間の`sky * fresnel`を加える順序に直した。これにより、値が1.0前後で横ばいの「台地」でも、Reinhardのなだらかな飽和特性(`L/(1+L)`)が正しく効き、中心が明るく周辺へ滑らかに減衰する形になった。

あわせて、"ハイライトの強さ"のデフォルト値も100%から10%へ引き下げた(このスライダーは元々「反射側の太陽を演出的に減衰できるノブ」として意図的に残していたもので、新しい規約でも値そのものの物理的な意味は変えていない)。トーンマップの修正だけでは、水平線際からしばらく1.0前後で横ばいという台地状の形状そのものは(Cox-Munkモデルが予測する現実の現象として)残るため、太さを写真に近づけるには全体の明るさを下げて「台地のうち本当に明るい部分だけ」がトーンマップ後に visible な白として残るようにする必要があった。

**教訓その3:** 「強すぎる/太すぎる」という指摘を受けて、最初はモデル(Beckmann分布の下限、幾何減衰項G)の物理的な妥当性ばかりを疑って手を加えたが、実際に効いたのは「線形HDRの合成順序」という実装レベルのバグだった。見た目のおかしさを物理モデルのせいだと決めつける前に、値がどの空間(線形かトーンマップ後か)でどう合成されているかを数値で追跡する(今回はNode.jsでの再現)ことが、遠回りに見えて最短だった。

**教訓その4:** この機能を実装している間に、並行して進んでいた別の変更(波のLODフェード、後述)が`WaveSample`に`unresolvedSlopeVar`というフィールドを追加し、コメントに「太陽ハイライトのきらめきの広がりに使う」という将来の用途まで明記した状態で先に取り込まれていた。両者を統合する際、当初実装していた「全波スペクトルの分散を使う」版のまま気づかずマージしていたら、幾何反射と統計モデルで同じ大スケールの傾きの寄与を二重に数えてしまうところだった。フィールドのコメントを読んで初めて用途に気づき、`unresolvedSlopeVar`を使う形に書き直した。関連する変更が同じ設計意図を別々に実装しつつあるとき、コミット履歴やコメントに残された「まだ使われていない伏線」を見落とさないことが、後から発覚する重複実装より安く済む。

**続報3: ユーザーに「実装全体にバグがないかチェックして」と依頼され、レビューで見つかった本物の欠陥 — 削除ではなく置き換えるつもりが、本当に削除されていた`sunDisc()`の反射側呼び出し**

上記の一連の調整が済んだ後、ユーザーから実装全体のレビューを依頼された。ファイル全体を読み直したところ、`sunGlitter()`を導入した箇所の複数のコメント(「解像できている波の輝きは既にreflect()による幾何学的な鏡面反射(sky\*fresnel、sunDisc())側に含まれている」など)が、水面反射側でも`sunDisc(reflDir, sunDir, ...)`を呼んでいる前提で書かれているのに、実際のコードでは`sunDisc(`の呼び出しが直視の太陽(`rayDir`を使う箇所)の1箇所しか無いことに気付いた(`grep -n "sunDisc("`で確認)。`sunGlitter()`を追加した際、本来は「解像できているスケールの輝き(`sunDisc(reflDir,...)`)にunresolved分の統計的きらめき(`sunGlitter()`)を**追加**する」つもりが、実際には`disc`変数の行をまるごと`glitter`の行に**置き換えて**しまい、`reflDir`が偶然ぴったり太陽を向いた場合の鋭いハイライトが完全に描画されなくなっていた。皮肉なことに、これは最初にCox-Munkモデルを導入する動機になった「ハイライトが疎で弱い」という問題の一部を、モデルではなく実装の欠落によって再発させていたことになる。`discHighlight = select(0.0, sunDisc(reflDir, sunDir, SUN_ANGULAR_RADIUS), reflValid) * sunVisibilityOf(sunDir)`として復元し、直視の太陽円盤と同じ0..1のLDRマスクとして(線形HDRの`glitterRadiance`とは別に)`sky * fresnel`と同じタイミングでトーンマップ後に加算した。

**教訓その5:** 既存の項を新しい項に「置き換える」実装をするときは、置き換え後にコメントを更新するだけでなく、コメントが主張している「別の項がその役割を担っている」という前提が実際にコード上で真かどうかを`grep`などで機械的に確認すべきだった。今回は3箇所のコメントが同じ誤った前提(「sunDisc()は反射側でも呼ばれている」)を共有したまま複数回のコミットを生き延びており、コメントの数が多いこと自体が「検証済み」という誤った安心感を生んでいた。ユーザーから「全体にバグがないかチェックして」という依頼を受けて初めて、ファイル全体を通しで読み直すという、個別の指摘に対応するだけでは行われない検証が働いた。

**続報4: 「波の高さを200%にしても均一に広がっているように見える。マクロ的な角度は考慮しているか?」— sunGlitter()の統計分布の基準面が世界の真上に固定されたままだった**

上記のバグ修正後、ユーザーから「波の高さスライダーを200%にしても、水面のきらめきが均一に広がって見える。波面のマクロ的な(=解像できている大スケールの)角度を統計モデルが考慮しているか?」と聞かれた。`sunGlitter()`の実装を見直すと、`let normal = vec3<f32>(0.0, 1.0, 0.0);`と、統計分布の基準面を常に世界座標の真上に固定していたことが分かった。これは元々「波が無い完全に平らな水面」を前提にしていた頃の名残りで、`unresolvedSlopeVar`(解像できていない微細な傾きの分散)だけを渡すよう直した際にも、そのままにしてしまっていた。

しかし`unresolvedSlopeVar`が表しているのは「解像できているこの局所面から、さらに細かい波がどれだけばらつくか」という**相対的な**ばらつきである。基準面を世界の真上に固定したままだと、「波の高さ」スライダーで解像できる大スケールの傾き自体をいくら急にしても、統計モデルの基準は動かないまま(sigma2がわずかに増えるだけ)なので、光る場所そのものは常に「波が無かったときと同じ、平らな鏡と同じ幾何学的な位置」から動かない。加えて、厳密な鏡面反射(`discHighlight`、`reflect()`が使う局所法線)と統計モデル(`sunGlitter()`、世界の真上固定)の基準面が一致していないため、同じ点でも「厳密な反射」と「統計的なきらめき」が別方向を向く矛盾も抱えていた。

修正は、`sunGlitter()`に`macroNormal`引数を追加し、呼び出し側から`waveSample.normal`(解像できているスケールの局所法線。discHighlightのreflect()が使っているのと同じもの)を渡すようにした。これは法線マップ+粗さのマイクロファセットシェーディングで、粗さ由来のNDFを常にワールド座標の真上ではなく法線マップ後の局所シェーディング法線を基準に評価するのと同じ考え方。修正後、波の高さ200%でヘッドレスレンダリングを比較すると、きらめきが波の稜線・谷に沿って明瞭に分裂した点状のパターンになり、20%のときの一様でぼんやりした帯とは明確に異なる見た目になることを確認した。

**教訓その6:** `unresolvedSlopeVar`の「二重計上を避ける」修正(教訓その4)をした際、分散という**大きさ**の整合性には注意を払ったが、その分散が「何を基準にしたばらつきか」という**基準面**の整合性を見落としていた。同じ関数が複数の物理量(分散と基準面)を扱っている場合、片方だけ直して満足せず、もう片方が古い前提(この場合は「水面はまだ平らだった」)のまま残っていないかを確認する必要があった。この見落としに気づけたのは、ユーザーが「均一に見える」という見た目の違和感を「マクロ的な角度を考慮しているか」という、実装の具体的な設計判断を問う形の質問に翻訳してくれたからで、見た目の指摘だけでは同じ気付きに至るのに時間がかかっていた可能性がある。

**続報5: 基準面の修正の直後、「中距離以遠で波の高さをしっかりと考慮していない影響がわかるようになってきてしまいました」— レイマーチングのステップ幅も振幅を考慮していなかった**

`sunGlitter()`を局所法線基準に直した直後、ユーザーから中距離以遠のきらめきの乱れを指摘された。これも同じ根っこの問題で、`intersectWater()`の遠距離側ステップ幅`TARGET_FAR_STEP`(1.5m)が波長だけを基準に決められていて、「波の高さ」スライダー(`ampScale`)では一切スケールしていなかった。

Nyquist的な波長基準のステップ幅選定自体は、波の空間周波数(=山と谷の間隔)を正しく捉えるという意味では振幅に依存しない一見正しい理屈に見える。しかし実際にレイがテストしているのは「波の高さフィールドの零点」ではなく「レイ(ほぼ水平線に近い浅い角度で近づく直線)と波の高さフィールドの交差」であり、振幅が大きいほど波面がレイの軌跡の上下に大きく振れるため、レイが波の振幅の範囲内に長く「捕まった」状態が続く区間が増え、1ステップ内でレイと波面が複数回交差する(=`f0`と`f1`の符号だけを見る現在のアルゴリズムが交差そのものを取りこぼす)可能性が高くなる。波の高さの基準面修正(教訓その6)によって`sunGlitter()`が交点の局所法線に直結するようになったことで、この取りこぼしによる不正確な法線が、それまで気付かれなかった中距離以遠のきらめきのノイズとして初めて視覚化された。

修正は`WAVE_HEIGHT_MARGIN`と同じ「縮む方向には縮めず、既定値を下限にする」パターンを踏襲し、`let farStepTarget = TARGET_FAR_STEP / max(uniforms.waveParams.x, 1.0);`として、波の高さが100%を超えるぶんだけ目標ステップ幅を狭める(=遠距離側のステップ数を増やす)ようにした。波の高さ200%・描画距離150mでのヘッドレスレンダリングを修正前後で比較し、中距離帯の水面がなめらかになることを確認した。

**教訓その7:** 「波の高さ」のような1つのスライダーが、複数の独立した箇所(この場合は統計モデルの基準面`normal`と、交点探索のステップ幅`TARGET_FAR_STEP`)にそれぞれ別々に影響しうることを見落としていた。`WAVE_HEIGHT_MARGIN`は既に同じスライダーでスケールしていたのに、`TARGET_FAR_STEP`は「波長だけの関数だから振幅とは無関係」という一見もっともらしい理屈に足を取られ、同じスケーリングが必要なことに気づいていなかった。1つの変更が及ぼす影響の範囲を検討する際は、「この定数は本当にこのパラメータと無関係か」を機械的に(`grep`でそのパラメータの他の使用箇所を洗い出すなど)確認するのが安全だった。

### 海中の散乱: 単散乱の閉形式積分(水の色をハードコードしない)

当初、フレネル透過側(水面下から届く光)はカラーピッカーで選ぶ固定の「水の色」だった。これを、大気の`sky-precise.html`と同じ発想で「色は光学係数と散乱の物理から出す」方式に置き換えた。

モデルは「均質な水」(当初は無限に深い水。現在は後述の通り指定した深さで海底に切り替わる)。太陽の平行光が水面で屈折して差し込み、屈折した視線の経路上の各点で1回散乱して眼に届く(単散乱)。経路上の距離`s`の点は深さ`z = s·dy`(`dy`=屈折視線の下向き成分)にあり、そこへ届く太陽光の水中経路長は`z/sy`(`sy`=屈折太陽光の下向き成分)。消散係数`c`(吸収+散乱)による減衰を視線側・太陽側の両経路に掛けると、

```
L = Σ_i b_i·P_i(θ)·E_uw·∫[0,∞) exp(-c·s·(1 + dy/sy)) ds
  = Σ_i b_i·P_i(θ)·E_uw / (c·(1 + dy/sy))
```

と**解析的に厳密に積分できる**(`i`は分子散乱と粒子散乱で、係数`b_i`と位相関数`P_i`が異なる)。レイマーチも乱数も不要な決定的計算なので、このシェーダーの「蓄積バッファなし・1パス」の構成をそのまま維持できる。

光学係数は実測に基づく値を使い、色は一切ハードコードしない:

- **純水の吸収**(Pope & Fry 1997の代表値): 赤 0.45 ≫ 緑 0.063 ≫ 青 0.0092 (1/m)。「海が青い」のはこの吸収の波長依存が主因で、実際に濁り0%では深い青になる
- **純海水の分子散乱**: レイリー散乱的に青が強い。位相関数もレイリー
- **濁り(懸濁粒子)**: スライダーで粒子の散乱(ほぼ波長非依存、位相関数はPetzold測定相当の強い前方散乱 HG g≈0.924)と、クロロフィルの吸収(青と赤を吸収して緑を残す)を同時にスケールする。濁りを上げると水が自然に緑へ寄る

このほか、太陽光の水面透過(フレネル`1-F`)、水中から空気へ出るときの透過`1-F`と屈折率による放射輝度の圧縮`1/n²`、天空光(天頂輝度を代表値にした拡散光)の寄与の近似を含む。海中の散乱は線形HDRで計算し、`sky-precise.html`のresolveパスと同じReinhard+ガンマで表示空間へ変換してから、LDRのままの空の反射とフレネルで合成する(焼き込みと同じトーンカーブを通すことで明るさのスケールが揃う)。

### フレネル反射率をSchlick近似から厳密な式へ(屈折角を陽に使う)

当初、水面のフレネル反射率(反射と透過の配分)は入射角のコサインだけから求まるSchlick近似`F = F0 + (1-F0)(1-cosθᵢ)⁵`を使っていた。一方`waterRadiance()`内の光路計算(散乱の経路長・海底の模様の位置)は最初からSnellの法則による屈折角`θt`を`refract()`で求めて使っており、フレネル反射率だけがこの屈折角を使わない近似のままだった。

厳密な(無偏光、s偏光・p偏光を平均した)フレネル方程式

```
Rs = ((n1 cosθᵢ - n2 cosθt) / (n1 cosθᵢ + n2 cosθt))²
Rp = ((n1 cosθt - n2 cosθᵢ) / (n1 cosθt + n2 cosθᵢ))²
R  = (Rs + Rp) / 2
```

に置き換えた(`θt`はSnellの法則`n1 sinθᵢ = n2 sinθt`から陽に計算する)。垂直入射(`θᵢ=0`)ではSchlickの`F0`と厳密解((n2-n1)/(n2+n1))²は共に空気→水(n1=1, n2=1.33)で≈0.02と一致するが、中間〜浅い角度では値がずれる(例: 入射角75°でSchlick 0.239 vs 厳密解 0.211、85°でSchlick 0.641 vs 厳密解 0.583)。空気→水(n1<n2)では全反射は起こらないため実害は無いが、`sin²θt`を1.0でクランプして安全にしている。

### 海の深さと海底(平面、プロシージャルな砂地)

「海の深さ」スライダーで、深さ`depth`(0.5〜300m)にある平らな海底を指定できる。無限深度の閉形式積分を、屈折視線が海底に届くまでの経路長`sMax = depth/dy`で打ち切るだけで、積分は閉形式のまま保てる:

```
∫[0,sMax] exp(-k·s) ds = (1 - exp(-k·sMax)) / k     (k = c·(1 + dy/sy))
```

`depth`を大きくすると`exp`項が消えて元の無限深度の式に滑らかに収束する(50mも下ると実際、海底はほぼ見えず深い青一色になる)。これに海底(Lambertian)の寄与を加える:

```
L_floor = albedo/π × E_floor × exp(-c·sMax)
E_floor = E_uw·sy·exp(-c·depth/sy)  (直達光: 水平な床への斜入射cos=sy、深さ/syの経路減衰)
        + π·L_zenith·exp(-2·c·depth) (天空光: 拡散光の平均斜入射を経路2倍で近似)
```

浅瀬(1〜2m)では赤だけが吸収されて海底の砂色がエメラルド色に透けて見え、深くするにつれ緑→青と沈んでいく、実際の浅い海と同じ色の変化が光学係数だけから出る。

海底のアルベドはプロシージャルな砂地の模様: 波長約0.8mの砂紋(位相を低周波バリューノイズで揺らして自然にうねらせる)+低周波ノイズの暗い斑(海草・岩場)+細かい粒感。遠方では模様の周波数が画素間隔を超えてモアレになるため、高周波の成分ほど早く距離フェードで平均色に落とす(`ocean-gerstner.html`の白波の距離フェードと同じ考え方)。模様の位置は屈折した視線が実際に海底へ達する座標(波面との交点+その点の波の法線で屈折した方向×`sMax`)で決めているので、深さを変えると屈折による見かけの位置ずれも正しく変化する。後述の通りガーストナー波の導入後は交点ごとに法線が変わるため、この屈折方向も波打って変化し、海底の模様が波にあわせて揺らめいて見える(コースティクスそのもの、すなわち隣接する波面の法線の収束・発散による光の集中は、単散乱を1点の局所法線だけで代表させる近似のため再現していない)。

実装時の余談: 海底の暗い斑の変数名を`patch`にしたところ`'patch' is a reserved keyword`でコンパイルエラーになった。`target`と同じWGSL予約語の落とし穴(README末尾参照)。

**既知の簡略化:** 多重散乱は含まない(散乱アルベド`b/c`が大きい濁った水ほど明るさを過小評価する)。太陽光の大気透過(夕方の赤み)は`sunVisibilityOf`の減光のみで近似。海底や浮遊物は無い。

### 表示空間のまま合成する(HDRへの逆変換をしない)

`ocean-gerstner.html`の環境マップ利用では、蓄積バッファがHDRの線形空間であることに合わせて、インポートしたLDR PNGをガンマ2.2+Reinhardの逆変換でHDRに復元してから使っている。`ocean-mirror.html`はHDR蓄積バッファを持たず(トーンマップし直す先が無い)、`sky-precise.html`自身の表示パスが焼き込みテクスチャをそのまま出力しているのと同じように、空の反射はLDRのまま合成して直接キャンバスへ出力する(海中の散乱だけは上記の通り線形HDRで計算してから同じトーンカーブで表示空間へ持ち込む。天空光の寄与を求めるための天頂輝度だけは逆変換で概算のHDRに戻して使う)。同じequirectangularテクスチャでも、それを使う側のパイプラインの性質(HDR蓄積があるか無いか)によって適切な合成空間が変わる、という一例になっている。

### 低い太陽で出た「黒い網点」: 微視的法線による太陽透過のゲート+点サンプリングのエイリアシング

朝夕(太陽高度が低いとき)に水面全体へ暗い網点状の斑が出た。フォールバック箇所の強調表示をONにしてもマゼンタにならないことから、交点なし・反射尽きのフォールバックではなく**有効なパスが計算した正規の色**であることがまず確定。計装モード(合成の各成分を単独表示するデバッグ分岐)で切り分けると、空の反射項には無く海中項(`waterRadiance()`)に完全に再現され、海底の模様を消しても残り、最終的に「太陽光の水面透過率 `1-F(dot(sunDir, normal))` のグレースケール表示」が黒点の分布と完全一致した。太陽高度を変えると黒点画素の割合が 8時9.4%・12時0.3%・16時11.0% と変化することも裏付けになった。

原因は2つの合成: (1)水柱・海底に差し込む太陽光の透過率を、視線が水面に入った1点の微視的ファセット(最小4cmの波)の法線でゲートしていたため、太陽から背いたファセットで透過が二値的に0へ落ち、水柱と海底の照明が同時に消えていた。実際には水中の各散乱点へ届く光は別の広い水面から入るので、この繋ぎ方は物理的に誤り。(2)遠方では4cm波はピクセルフットプリントより遥かに小さく、固定2x2サブサンプルの点サンプリングが「ほぼ0/ほぼ1」の二値を拾って硬い網点に量子化していた(実カメラの1画素はファセット傾斜分布の平均=滑らかな値を記録する)。

修正も2段: (1)太陽照明側(透過フレネルと屈折太陽光`sy`)だけ、うねりスケール(波長1m以上)の波で作った低周波法線`normalSun`で評価する。視線側(反射・屈折・視線フレネル)は従来通り局所法線のままなので、鏡面反射のシャープさと海底模様の屈折歪みは保たれる。(2)波のLOD: 波長λの波が画素で解像できなくなる距離(ナイキスト条件からおよそ`45√λ` m)で、その波を振幅ごと高さ・法線の両方からフェードアウトする。遠方の海面は滑らかなうねりだけに収束し、「未解像の微細構造は平均されて見えない」という実写の見え方と一致する。修正後、黒点画素は8時・12時で0%、16時で2%になった(残りは波と一緒に動く滑らかな陰影で、うねりが太陽から背いた斜面で透過率が下がる物理的な変調)。あわせて、海底模様のモアレ対策フェードに渡す距離が水中経路長だけ(高々8m程度)でカメラからの距離が入っておらず実質無効だった問題も、空気中+水中の合計経路長を渡すよう修正した。

## 高精度な空(多重散乱パストレーシング)の概要

`shaders/sky-precise.html`は、`analytic-sky.html`の1ピクセル1回の解析的近似とは対照的に、大気を球殻状の不均質な媒質としてモンテカルロ・ボリュームパストレーシングし、多重散乱まで含めて物理的に(遅いが正確に)計算する参照実装です。Cornell Boxのパストレーサーと同じ「フレームを重ねて時間積分でノイズを収束させる」方式を採用しています。

### 物理モデル

惑星半径6360km・大気の厚さ60kmという球殻ジオメトリの上で、高度に応じて指数関数的に減衰するレイリー散乱・ミー散乱と、成層圏付近(高度25km付近)で釣り鐘状に分布するオゾン吸収を重ね合わせています。係数はBruneton & Neyretのprecomputed atmospheric scatteringモデルで広く使われている値を採用しました。視線方向のレイをレイマーチしながら、各ステップの消散率から次の衝突点までの距離を指数分布で確率的にサンプリングし(自由行程サンプリング)、衝突が起きたらレイリー散乱・ミー散乱・吸収のどれが起きたかを輝度加重で確率的に選び、Next Event Estimation(衝突点から太陽方向への直接寄与を明示的に加算)と、選んだ位相関数からの重点的サンプリングによる次方向への継続を繰り返します。

### equirectangularテクスチャへの焼き込み

方向(=equirectangularテクスチャのテクセル)ごとに独立にこのパストレーシングを行い、HDRの蓄積バッファ(rgba32float、ping-pong)に「フレームを重ねるほど収束する時間平均」として書き込みます。表示・PNGエクスポート・PNGインポートは、この蓄積結果をトーンマップ+ガンマ補正した8bitのequirectangularテクスチャを介して行うため、収束の途中でもいつでもエクスポートでき、インポートした画像は蓄積を再開するまで固定表示されます。太陽の円盤はこのテクスチャに焼き込まず、表示パスが時刻から解析的に重ねて描きます(後述「太陽円盤は焼き込まず、サンプリング時に解析的に描く」)。

### 検証結果: 夕焼けの赤みが物理計算から自然に出現する

太陽の時刻を地平線際(既存の太陽軌道の式で言うとちょうど18:00が日没)に近づけてレンダリングしたところ、色を一切ハードコードしていないにもかかわらず、太陽付近だけがオレンジ~金色に色づき、そこから離れるほど紫~濃い青の夜空へ滑らかに移り変わる、実際の夕焼け写真に近いグラデーションが得られました。これは「低い太陽ほど、光が地表付近の濃い大気を長い距離通過するため青が先に散乱で失われ、赤だけが残る」という現象が、球殻ジオメトリと高度依存の消散係数から自然に導かれた結果であり、`analytic-sky.html`のように夕焼け色を式の形で明示的に作り込んだものではありません。

### 既知の簡略化: 直接見える太陽円盤・背景のスペクトル減衰

自由行程サンプリング(次の衝突点までの距離を決める確率)には、実装を単純化するためRGB各chの消散係数を輝度平均した単一のスカラー値を使っています。散乱イベントが実際に起きた場合の色(NEEでの太陽方向への寄与や、スループットへの反射率の乗算)は各chの値を正しく使っているため、画面に表示される色の大部分(散乱光による空のグラデーション)は正しくスペクトル依存です。

太陽本体の円盤については、後述の「太陽円盤の明るさ」の修正と合わせて、この輝度平均によるスペクトル不正確性も解消しています(NEEと同じ決定的な数値積分に置き換えたため)。星や夜空の背景色にはまだこの簡略化が残っていますが、視覚的な寄与が小さいため実害はありません。

### 太陽円盤の明るさが正しく表現できていなかった問題(照度と輝度の混同)

ユーザーから「この太陽円盤は大気散乱の計算とは別に個別で描画しているのか」という質問を受けて調べたところ、円盤は`traceSkyPrecise()`内で「カメラの主レイが一度も散乱されずに大気圏外まで抜けた(`collided == false`)」場合にだけ加算される、散乱計算とは独立した加算項であることが分かった。さらに深掘りする過程で、この円盤の明るさが物理的に大きく誤っていたことも発覚した。

NEEの散乱寄与に使っている`SUN_INTENSITY`は「太陽から降り注ぐ照度(irradiance, 単位面積あたりのエネルギー)」に相当する量で、位相関数によって単位立体角あたりの散乱光に変換されるため、そのままの大きさで正しい。しかし太陽円盤を直接見たときに見えるべきなのは「輝度(radiance, 単位立体角あたりの明るさ)」であり、両者は太陽の見かけの立体角Ω(角半径0.02ラジアンとして`2π(1-cosΩ)`≈1.3×10⁻³ sr、天球全体の1/5000程度しかない)を使って`L = E / Ω`という関係で変換する必要がある。修正前のコードは`SUN_INTENSITY`を円盤にもそのまま使っており、変換を怠っていたため、円盤は「空の散乱光と同程度の明るさ」にしかなっていなかった(実際の太陽は空よりも桁違いに明るいはずだった)。Reinhardトーンマップ(`color/(color+1)`)は1を大きく超えるHDR値をどれもほぼ同じ白飽和に潰してしまうため、この誤りは画面の見た目だけでは気づけなかった。

修正では、円盤の輝度を`SUN_INTENSITY / Ω`として正しく計算し直した。

さらに、ユーザーから「太陽を桁違いに明るくすると暗い部分の精度が悪化しないか」という追加の指摘を受けた。調べたところ、明るさの補正自体は無関係なテクセルの数値精度には影響しない(蓄積バッファはequirectangularテクセルごとに完全に独立したrunning averageであり、共有の指数部や自動露出のような仕組みも無いため)一方で、**円盤自体の縁**には実際にリスクがあった。旧実装は「衝突しなかった(自由行程サンプリングの結果)」という確率変数(Bernoulli)で円盤の寄与の有無を決めていたため、円盤の境界にちょうど掛かるテクセル/ピクセル(equirect焼き込み・direct計算どちらもサブテクセル/サブピクセルジッタでAAしているため境界付近は毎フレーム円盤の内外を行き来する)は「0」と「円盤の値」を確率的に行き来することになる。明るさを立体角補正で約1000倍にすると、この二値のばらつきがそのまま約1000倍の分散になり、円盤の縁の収束が大幅に悪化するおそれがあった。

対策として、円盤の寄与をNEEと同じ考え方で書き直した。自由行程サンプリングの確率的な結果(`collided`)に関係なく、`bounce == 0`の毎サンプルで無条件に、視線方向の透過率を決定的な数値積分(`computeSunTransmittance()`。既にNEEで太陽方向への透過率計算に使っていたのと全く同じ関数で、方向を太陽方向ではなく視線方向に差し替えるだけで、地面に遮られていれば内部で自動的に0を返す)で計算して加算するようにした。これによりノイズの発生源だった確率的な二値判定そのものをこの項の計算から切り離し、円盤の明るさをどれだけ大きくしてもその分だけノイズが悪化することは無くなった(むしろ旧実装より低ノイズになっているはずである)。副次的に、この決定的な透過率計算はRGB各chを個別に扱うため、前項で触れていた「太陽円盤のスペクトル不正確性」という既知の簡略化も一緒に解消された。

**教訓:** ある実装がエラーも出さず一見自然に動いていても、物理量の次元(照度 vs 輝度、確率的推定 vs 決定的推定)を混同していると気づきにくい形で誤りうる。今回は「平らな鏡面なら太陽の形は歪まないはず」「太陽を明るくしたら他の部分の精度が落ちないか」という、ユーザーの2つの物理的な直感からの質問がどちらも正しい懸念で、実装を見直す決め手になった。

### 太陽円盤は焼き込まず、サンプリング時に解析的に描く

上記の明るさ・分散の修正を経てもなお、太陽円盤をequirectangularテクスチャに焼き込む方式には構造的な問題が残っていた。

- **LDR化で輝度が失われる**: エクスポート/表示に使う8bitテクスチャはReinhardトーンマップ済みで、円盤の輝度(`SUN_INTENSITY/Ω`≈1.2万)は最大でも1.0にクリップされる。利用側(`ocean-mirror.html`の反射など)でフレネル係数(垂直入射で0.02)を掛けると「白い空と同じ明るさ」まで減衰した太陽は透過色に埋もれ、「反射しても桁違いに明るい」という太陽の本質を原理的に再現できない
- **テクセル解像度の制約**: 角半径0.02ラジアンの円盤はequirect上で直径十数テクセルしかなく、bilinear補間で痩せて、参照する側の解像度・AAとの組み合わせ次第でチラつく。この対策のためだけに焼き込み解像度を2048x1024へ上げる(コスト4倍)必要があった

そこで方式を変え、**焼き込むのは滑らかな散乱光のみ**とし、太陽円盤は「画像+太陽の時刻」のペアを前提に、テクスチャを参照する全ての場所で時刻から解析的に描くことにした(円盤の位置は`hourToSunDir()`から決定的に求まるため、テクスチャに情報を持たせる必要がない)。具体的には:

- `traceSkyPrecise()`に`includeSunDisc`フラグを追加し、equirect焼き込みでは円盤を加算しない。検証用の直接計算モードだけは「最終的に見えるべき真値」の参照としてHDRのまま円盤込みで計算する
- `sky-precise.html`自身の表示パスが、レイ方向と太陽方向のなす角から円盤マスクを計算し、輝度`SUN_INTENSITY/Ω`×視線方向の透過率(決定的な数値積分。日没際の赤みと地面による遮蔽はここから出る)をトーンマップして重ねる。テクスチャがLDRなので厳密な`tonemap(空+円盤)`はできないが、円盤が見える場所では円盤輝度が空より桁違いに大きく`tonemap(空+円盤)≈tonemap(円盤)`となるため、`max(空, tonemap(円盤))`で合成する(円盤が空より暗い場合は自然に空へ沈む)
- 利用側の`ocean-mirror.html`は直視・反射の両方に同じ解析円盤を描き、`ocean-gerstner.html`はインポート空の`skyRadiance()`に解析空と同スケールの円盤項を加算する

古い(太陽入りの)PNGをインポートしても、解析円盤がmax合成で焼き込み円盤とほぼ同じ場所に重なるため、時刻スライダーが焼き込み時刻と合っていれば見た目はおおむね後方互換になる。焼き込み解像度は、太陽のためという当初の理由は消えたが、水平線付近のグラデーションのディテールのため2048x1024を維持している。

この方式は「画像+焼き込み時刻」のペアが揃って初めて成立する(時刻がずれると太陽だけ別の場所に描かれる)ため、時刻の受け渡しを手動のスライダー合わせに頼らず自動化している。エクスポート時に焼き込み時刻をPNGの`tEXt`チャンク(キーワード`sunHour`)として埋め込み(ファイル名にも`sky-equirect-1600.png`のように付与)、インポートする側(`sky-precise.html`自身・`ocean-mirror.html`・`ocean-gerstner.html`)はチャンクを走査して時刻を読み出し、スライダーへ自動反映する。`tEXt`チャンクの挿入はcanvas APIでは扱えないため、`toBlob()`が生成したPNGのバイト列のIENDチャンク直前に、PNG仕様のCRC-32を付けた自前のチャンクを挿入する形で実装している。時刻情報の無い古いPNGはスライダーを手動で合わせる従来の運用になる(ステータス表示で「時刻情報なし」と分かる)。

### 太陽の周りに見えた「楕円状の境界」の正体: バグではなく地平線増光(horizon brightening)

サンプル数を上げてノイズを収束させていくと、太陽の周りに(equirectangularテクスチャ上で)楕円状の不連続な境界のように見える模様が現れ、当初はバグ(多重散乱の打ち切りや自由行程サンプリングの精度不足)を疑って調査した。まず主レイのレイマーチングを固定16ステップの均等刻みから、カメラに近い側を密に・遠い側を粗く刻むu²の非均等刻み(24ステップ)に変更してみたが、境界の見た目は変化しなかった。

次に、Node.js側で単一散乱項だけを超高精度(400ステップ)な数値積分で再現し、太陽から見て「真下(地平線側)」方向にだけ角度をずらしながら明るさを計算したところ、太陽から離れるにつれて一旦暗くなった後、地平線に近づくにつれて再び明るくなる(非単調な谷ができる)ことが確認できた。これは実装のバグではなく、視線が水平に近づくほど大気中を通過する距離が長くなり、1点あたりの散乱効率は下がっても総量としては多くの光を拾う「地平線増光」という実在の物理現象だった。太陽近傍の明るさ(ミー散乱による太陽環)と、地平線そのものの明るさという、由来の異なる2つの明るい領域が谷を挟んで隣り合うことで、あたかも境界があるかのように見えていた。

**教訓:** 「サンプル数を増やすほど目立つ」「不自然に見える」という所見だけでは、それがバグによるものか、単に今まで気づいていなかった実在の物理現象がノイズの下から現れてきただけなのかを区別できない。今回は「境界を挟んで上下(天頂側・地平線側)で明るさの変化パターンが非対称かどうか」を確認したこと、および同じ現象を全く別の実装(独立したNode.js数値積分)で再現できるかを確認したことで、初めてバグ説を退けられた。レイマーチングの非均等刻み自体は数値精度の観点で正当な改善なのでそのまま残しているが、今回の「境界」の直接の原因ではなかった。

## 開発中に見つかった落とし穴

### 夕焼けの色が付かなかった問題(`shaders/analytic-sky.html`)

大気散乱シェーダーの最初の実装では、視線方向の光路長(水平線に近いほど大気を長く通る)だけを減衰の要因にしていた。式で書くと、

```
viewExtinction = exp(-viewOpticalDepth * (rayleighCoeff + mieCoeff) * scale)
inscatter = (rayleighCoeff*rayleighPhase + mieCoeff*miePhase) * viewExtinction
```

これだけだと、離れた方角(太陽から離れた斜め上とか)を見てもほぼ同じような灰色っぽい色にしかならず、正午の青空も夕焼けのオレンジも弱々しくにじんだような色にしかならなかった。

原因は、**太陽光そのものが地表に届くまでに大気を通ってきた分の減衰**をモデルに入れていなかったこと。夕焼けが赤くなるのは、太陽が低い角度にあるときは太陽光が長い距離大気中を進んでくる間に青い波長がRayleigh散乱で強く散乱されて抜け落ち、赤系の波長だけが生き残って届くから。視線方向の減衰だけでは「見ている方向によって色が変わる」効果しか再現できず、「太陽の高さによって空全体の色調が変わる」効果が抜けていた。

修正として、太陽の仰角(`sunDir.y`)から求めた光路長でも同様に波長依存の減衰(`sunTransmittance`)を計算し、視線方向の減衰と掛け合わせるようにした。

```
sunTransmittance = exp(-sunOpticalDepth * (rayleighCoeff + mieCoeff) * scale)
inscatter = (rayleighCoeff*rayleighPhase + mieCoeff*miePhase) * viewExtinction * sunTransmittance
```

これで太陽が水平線に近づくと`sunTransmittance`の青チャンネルだけが大きく減衰し、赤チャンネルが相対的に生き残ることで、ちゃんとオレンジ色の夕焼けになった。太陽本体(円盤)の色も`sunTransmittance`で染めるようにしたので、正午は白っぽく、夕方は赤っぽく見えるようになっている。

オフスクリーンでピクセル値を読み出して検証していたが、修正前は「太陽から離れた斜め方向」と「太陽方向(円盤を避けた位置)」のRGB比がほとんど同じ(せいぜい数%の差)だったのに対し、修正後は明確にR>G>Bとなり(例: 夕焼けで`[116, 86, 44]`)、数値上でも違いが確認できた。

### WGSLの予約語との衝突

カメラの注視点を表す変数名として`target`を使ったところ、`error: 'target' is a reserved keyword`でシェーダーのコンパイルに失敗した。WGSLにはこの手の予約語がいくつかあり、うっかり変数名にすると分かりにくいエラーにはならず一発でコンパイルエラーになるので気づきやすいが、変数名の選択には注意が必要(このプロジェクトでは`lookAt`のような別名に置き換えて回避した)。

### ヘッドレス環境でのcanvas読み出しの制限

このセッションのようなGPUパススルーの無いサンドボックス環境(swiftshaderによるソフトウェアレンダリング)では、WebGPUの初期化やオフスクリーンテクスチャへのレンダリング自体は問題なく動くが、実際に画面に表示される`<canvas>`の内容を`drawImage`やスクリーンショットで読み出すと常に透明/黒になった。そのため、動作確認は「画面のcanvasを直接読み出す」のではなく、「同じシェーダーロジックを使ってオフスクリーンのテクスチャに描画し、`copyTextureToBuffer`で直接ピクセル値を読み出す」という方法に切り替えて行っている。実機のブラウザで見た目を確認する場合はこの制限は無関係。

### シェーダーのループ本数を増やしたら再現性100%でハングした(`shaders/ocean-mirror.html`)

波を表現するガーストナー波の本数(`NUM_WAVES`)を8→20に増やしたところ、ヘッドレス検証の`toDataURL()`呼び出しがCPU使用率ほぼ0のまま何十秒経っても返ってこなくなった。描画距離やビューポートを縮小しても再現したため負荷の問題ではないと判断し、`NUM_WAVES`を2分探索したところ、16は必ず成功し17は必ずハングするという明確な境界が見つかった(swiftshaderのシェーダーコンパイラかドライバの何らかの上限に当たっていると見られるが、WGSL側から根本原因は特定できなかった)。実機の本物のGPUでは発生しない可能性もあるが、検証できない環境まで含めて安全側に倒し、実証済みの16に留めた。

これとは別に、この調査の途中で「50秒待って応答が無い」と判断してNUM_WAVES=16(実は正常に動くはずの値)のテストプロセスを一度強制終了させてしまったが、実際には裏で正常終了していたことが後から判明した(タイムアウトを100秒に伸ばして再実行したら普通に完了した)。ヘッドレス環境(特にソフトウェアレンダリング)はシェーダーのコンパイルや初回描画が実機よりかなり遅いことがあるため、**「反応が無い=壊れている」と即断せず、まず待機時間を伸ばして再現するか確認する**べきだった。

### パラメータチューニングを、実際のカメラ/太陽の向きと無関係なテスト値でやってしまった

空の色(`shaders/analytic-sky.html`)を「もっと空色っぽく」調整したとき、最初は自分で適当に選んだ視線方向・太陽方向のベクトル(例:「太陽から少し離れた斜め上」)を使って`skyColor()`だけを単体テストし、その出力が目標の色(`[135, 206, 235]`)に近づくようRayleigh/Mie係数をNode.js上で数値的に最適化した。単体テストとしては目標色にほぼ一致する結果が出たが、実際のシェーダー(固定カメラの向き+太陽が時間で動く軌道の式)に同じ係数を適用してオフスクリーンで検証し直すと、画面中央付近が真っ白に近い色になってしまい、テストで見ていた色とはまったく違う結果になった。

原因は、テストに使った視線方向・太陽方向のベクトルが、実際のカメラの向き(`forward = normalize(vec3(0, 0.35, -1))`)や太陽の軌道式(`sunDir = normalize(vec3(cos(sunAngle)*0.35, sin(sunAngle), -0.9))`)とかけ離れた、自分で適当に決めた値だったこと。特に「正午」のテストでは太陽の仰角を`sunDir.y ≈ 1`(真上)と仮定していたが、実際の軌道式では太陽が単純な円運動ではなく`z`成分が固定なので、正規化後の最大仰角は`0.74`程度にしかならない。テスト用のベクトルと実際にレンダリングされる視線・太陽の位置関係(特に`dot(視線, 太陽方向)`の値)がずれていたため、そのテストに対して最適化したパラメータは実際の画面には通用しなかった。

対策として、実際のカメラの基底ベクトル(`forward`/`right`/`up`)と太陽の軌道式をNode.js側にもそのまま複製し、実際に画面に表示される方向(画面中央など)でパラメータを検証し直した。**シェーダーの一部分だけを単体テストする場合、その入力(視線方向や太陽方向など)が実際のレンダリングで発生する値の範囲を代表しているか確認しないと、テストで良い結果が出てもそれは実物とは無関係な結果になりうる**、という教訓。

### `textureSample`をピクセルごとに分岐する処理の中で呼んでコンパイルエラー(`shaders/sky-precise.html`)

equirectangularテクスチャに焼き込んだ空とその場で直接計算した空を画面の左右半分ずつ表示して見比べる検証用シェーダーで、`if (splitX < 0.5) { ... } else { let sampled = textureSample(...); }`のように`@builtin(position)`(ピクセルごとに異なる非uniformな値)に依存する分岐の中で`textureSample`を呼んだところ、`'textureSample' must only be called from uniform control flow`というコンパイルエラーになった。`textureSample`はミップマップレベル選択のために近傍ピクセルとの微分(dpdx/dpdy)を暗黙に計算するため、ワープ内の全ピクセルが同じ分岐を通ることが保証されない箇所では呼べない、というWGSL(というよりGPUのderivative命令一般)の制約。今回はミップマップ自体不要なので、LODを明示的に指定する`textureSampleLevel(tex, sampler, uv, 0.0)`に置き換えることで解決した。任意の分岐の中でテクスチャサンプリングしたい場合は`textureSampleLevel`を使うのが安全。

### サンプル数を増やすほどノイズが悪化した問題(NEEとBSDF/位相関数サンプリングの二重カウント、`shaders/sky-precise.html`)

多重散乱パストレーシングの実装当初、太陽の時刻を10:00でエクスポートしたPNGを見た限りでは滑らかに収束していたが、ユーザーが自身の実機で1万サンプルまで蓄積したところ、むしろサンプル数を増やすほどホタル(fireflies)状のノイズが目立つという逆転現象が報告された。原因は、各衝突点で太陽方向への直接寄与をNext Event Estimation(NEE)で明示的に計算しているのに加え、そのまま位相関数から確率的にサンプリングした続きの方向がレイの終端(大気圏外)まで散乱なしに到達し、かつたまたま太陽の位置と一致した場合にも「太陽円盤を直接見た」として同じ物理的寄与を二重に加算していたこと。太陽は角度的に立体角にして全天球の約18万分の1しかないため、この「たまたま一致する」事象は極めて稀だが、発生したときの寄与は(NEEの寄与とほぼ同じ大きさなので)相対的に大きく、これが典型的なホタルノイズの原因になる。サンプル数を増やすほど、画面のどこかでこの稀なイベントが新たに発生する回数も増えるため、かえってノイズが目立って見えていた。

修正は、散乱後(まだ一度もぶつかっていないカメラの主レイではない)の経路については、位相関数サンプリングによる太陽への直接ヒットの加算を完全に無効化し、太陽への直接光はNEEだけに一本化するというもの。これはNEEが常に(確率1で)評価されている光源に対するMultiple Importance Samplingの特殊ケースにあたり、太陽のように立体角が極小の光源ではBSDF/位相関数サンプリング側の重みがほぼゼロに収束するため、期待値(明るさ)を変えずに分散だけを大きく減らせる。**光源が常にNEEでサンプリングされているなら、ランダムに歩いた経路がその同じ光源に偶然到達したときの寄与を、NEEと別に無条件で足し合わせてはいけない**、という教訓。

### 2000サンプルを超えたあたりから縞(バンディング)が現れる問題(f16蓄積バッファの精度限界、`shaders/sky-precise.html`)

サンプル数が少ないうちはきれいなのに、2000サンプルを超えたあたりから空のグラデーションに沿った縞模様が浮かび上がってくる、という現象が報告された。レイマーチングのステップ数・ステップ幅の偏りをどう変えても結果が全く変わらなかったことから、積分(レンダリング方程式の計算)側ではなく蓄積側に原因があると当たりを付けた。

原因は、時間積分の蓄積バッファに使っていた`rgba16float`(半精度浮動小数点)の精度限界。累積平均の更新式`newAvg = prev + (sample - prev) / n`は、nが大きくなるほど1回あたりの更新量が小さくなるが、f16の相対量子化幅は約2⁻¹¹(≈0.05%)しかないため、**n≈2000あたりで更新量が量子化幅を下回り、丸めで更新が消えて平均が凍結し始める**。凍結した値はf16で表現可能な離散的な値に吸着するため、本来滑らかなグラデーションが等高線状の縞になる。サンプル数が少ないうちに見えなかったのは、周囲のモンテカルロノイズが縞より大きくて隠れていたからで、ノイズが収束するにつれて縞だけが浮かび上がってきた。Node.js上でf16/f64それぞれの蓄積を並行シミュレーションしたところ、f64の誤差はn=6000まで単調に減り続けるのに対し、f16の誤差はちょうどn≈2000で下げ止まって凍結する、という報告と完全に一致する挙動が数値的に再現できた。

修正は蓄積バッファを`rgba32float`に変更するだけ(f32は仮数24bitなので、この方式の凍結が問題になるのは数百万サンプル以降)。**「サンプル数を増やすと悪化する」系の症状でも、原因は必ずしもサンプリング理論(前項のNEE二重カウントのような)とは限らず、蓄積ストレージの数値精度のこともある**。running average方式は総和を保持しないためオーバーフローには強いが、代わりに「更新量が徐々に小さくなる」性質があり、低精度フォーマットとの相性が悪い、という教訓。なおCornell Boxのシェーダーも同じ`rgba16float`蓄積を使っているため、同様に数千サンプルで収束が頭打ちになるが、表示がトーンマップ+8bit canvas経由であり、かつ空のような超低周波の滑らかなグラデーションが画面の大半を占めるわけではないため、実害が視認されにくい(必要になったら同じ修正を適用する)。海のシェーダーは焼き込み空の環境マップ導入に伴い長時間の蓄積が現実的になったため、`rgba32float`に修正済み。

### 原因究明の前に症状を隠す対処(ぼかし)をしてしまった

太陽の周りの「境界」問題(前述の地平線増光)を調査していた際、原因を特定しきれない段階で「最終用途(海面反射の環境マップ)ではぼやけるから実害はない」と判断し、解決パスに空間ガウスぼかしを1パス追加して見た目だけ滑らかにする対処を先に入れてしまった。ぼかしは症状(縞・ノイズ)を確かに隠すが、原因が分からないまま隠すと(1)本物のバグを覆い隠して後段の検証を全て信用できなくする、(2)本物の物理現象(地平線増光)まで一緒に消してしまう、という二重の害がある。後にユーザーの指摘でぼかしをrevertして原因究明に切り替え、最終的に「地平線増光は実在の現象」「縞はf16蓄積の精度限界」という2つの真因に到達できた。**再現条件が明確な視覚アーティファクトに対して、原因不明のままローパスフィルタを掛けるのは調査の放棄と同じ**、という教訓。

また、このrevertの際に副次的な失敗も起きた。ぼかしのコミットに(ぼかしとは無関係な)NEE二重カウントのREADME記述を同梱してしまっていたため、`git revert`でぼかしを取り消したときにドキュメントまで巻き添えで消えてしまい、しばらく気づかなかった。**1つのコミットには1つの論理的変更だけを入れる(無関係なドキュメント追記を実装コミットに混ぜない)**という基本の重要性を、revertして初めて実感した。

### もっともらしい仮説をいくつも実装したが、どれも症状に効いていなかった(バンディング調査)

f16蓄積が真因だったバンディング問題では、真因に到達するまでに複数の「もっともらしい仮説」を実装・検証して外した。(1)多重散乱の打ち切り(最大散乱回数)不足説 → 4回と8回で比較して差がなく棄却。(2)HG位相関数のfloat精度不安定説(外部からの指摘) → 分母のヒートマップを描いて滑らかであることを確認して棄却。(3)水平線付近のレイマーチ解像度不足説 → ステップをu²の非均等刻みに変更したが症状は全く変わらず棄却(ただし数値精度の改善としては正当なので残した)。(4)equirectテクスチャの解像度・補間説 → テクスチャを介さない直接計算モードを実装して見た目が改善したため一時はこれを結論としかけたが、実際は「解像度が上がって縞が細かくなり見えにくくなった」だけで、直接計算でも蓄積が進めば同じ縞が出ることがユーザーの実機で判明した。

決定打になったのは、ユーザーが行った「パラメータを変えて結果の絶対値を比較する」という検証で、レイマーチのステップ数・幅をどう変えても結果が全く変わらないという事実だった。これは症状の発生源が積分計算より後段(蓄積側)にあることを強く示唆し、そこからf16の量子化幅と「n≈2000で更新が消える」という閾値の一致に気づけた。**仮説を実装したら、その変更が症状の大きさを変えるかどうかを必ずbefore/afterで確認する。変化しないなら、その仮説はどれだけもっともらしくても間違っている**、という教訓。

### 強調表示に「引っかからない」アーティファクトは、成分分解の計装で絞り込む(`shaders/ocean-mirror.html`)

水面の黒い網点は、フォールバック箇所の強調表示をONにしても色が変わらなかった。この「変わらない」こと自体が重要な情報で、既知のフォールバック経路(交点なし・反射尽き)は除外され、有効パスの計算結果に犯人がいることが確定する。そこからは、最終色を構成する項(空の反射項/海中散乱項/海底項/フレネル/太陽透過率…)を1つずつ単独でキャンバスに出す計装モードをデバッグ用uniformで足し、症状のパターンと一致する項を二分探索的に絞り込むのが決定打だった(「海底の模様を一様な色に置換しても残る→模様ではない」「太陽透過率のグレースケール表示が黒点分布と完全一致→これが犯人」)。**「どの項が犯人か」を推測で議論する前に、各項を単独表示する計装を書く方が速くて確実**。あわせて、症状がパラメータ(この件では太陽高度)にどう依存するかを数値で測る(黒点画素の割合: 8時9.4%・12時0.3%・16時11.0%)と、仮説の裏付けが定量的になる。

### 微視的な法線で体積照明をゲートすると、グレージング角で二値化してエイリアシングになる(`shaders/ocean-mirror.html`)

フレネル式そのものは正しくても、「どの法線に適用するか」を誤ると破綻する、という教訓。水中の散乱や海底に届く太陽光は視線の入射点とは別の広い水面から入ってくるので、その透過率を視線入射点の微視的ファセット1枚の法線で決めてはいけない(太陽が低いと`dot(sunDir, normal)≦0`のファセットで透過が0/1の二値になり、ピクセルフットプリントより小さい波がそれを網点にエイリアスする)。鏡面反射のように「まさにその1点」で起きる現象には局所法線を、体積や遠くの面への照明のように「広い面の平均」で決まる現象には低周波法線(または傾斜分布での平均)を使う。エネルギー保存の観点でも、透過が0のファセットは反射率1で空を明るく映すはずで、真っ黒に沈む見た目はその時点で物理と矛盾している。

### equirectangular画像をピクセル空間のまま放射状に分析してしまった

太陽を中心とした「境界」の半径方向の輝度プロファイルを調べる際、最初はequirectangular画像上でのピクセル距離でリング状に平均を取っていた。しかしequirectangular(緯度経度)図法では、赤道から離れた位置を中心とするピクセル空間上の円は、球面上の実際の角度としては円ではなく大きく歪む。このため最初の分析は「太陽からの角度」という本来見たい軸に対して別の構造(地平線との距離)を混ぜ込んでおり、プロファイルの解釈を誤らせかけた。ピクセルごとに方向ベクトルを復元して`dot(方向, 太陽方向)`から真の球面角度を計算し直し、さらにリングを「太陽より天頂側」と「地平線側」に分けて集計したことで、天頂側は完全に滑らか・地平線側だけに谷がある、という非対称性が初めて見え、地平線増光(太陽からの角度ではなく仰角に依存する現象)の発見につながった。**投影された画像の上で幾何学的な集計をするときは、投影の歪みを補正してから行う。また「対称なはず」の軸で分けて集計してみると、現象がどの変数に依存しているかを切り分けられる**、という教訓。
