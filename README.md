# AI駆動でWebGPUプログラミング

ビルド不要のバニラJS/HTMLで作ったWebGPUシェーダープレイグラウンドです。`index.html`を開くとシェーダー一覧のホームページが表示され、各リンクから個別のシェーダーページに遷移します。

**デモ:** https://shugo15.github.io/webgpu-with-ai/

## 構成

- `index.html` — シェーダー一覧のホームページ
- `shaders/rotating-hue-gradient.html` — 画面中心を軸に回転しながら色相が変化するグラデーション
- `shaders/cornell-box-pathtracing.html` — SDFレイマーチング + パストレーシングによるCornell Box
- `shaders/cornell-box-specular.html` — Cornell Boxに完全鏡面反射・スペキュラ(グロッシー反射)の球を追加したもの
- `shaders/analytic-sky.html` — レイマーチングもモンテカルロ積分も使わない、解析的な高速近似による空(大気散乱)

各シェーダーページは1つの`.html`ファイルに完結しており（WGSLはJS内の文字列として埋め込み）、サーバー不要でファイルを直接ブラウザで開けば動作します。

## Cornell Box レイマーチングの概要

`shaders/cornell-box-pathtracing.html`は、SDF（符号付き距離関数）によるレイマーチングと、簡易的なパストレーシングを組み合わせてCornell Boxを描画しています。

### SDF（符号付き距離関数）とレイマーチング

`map(p)`関数が、空間上の任意の点`p`から「最も近いオブジェクトの表面までの距離」を返すSDFです。壁や天井・床は平面までの垂直距離、内部の箱は`sdBox()`という定番の公式で距離を計算し、`opU()`（複数の距離のうち最小のものを選ぶ処理）で全オブジェクトを1つの距離関数にまとめています。

`raymarch()`はこのSDFを使ってレイを飛ばす手法（スフィアトレーシング）です。レイの現在位置から`map(p)`で最寄りの表面までの距離を求め、その距離ぶんだけレイを前進させる、を繰り返します。距離がほぼ0になったら表面にヒットしたと判定します。ポリゴンとの解析的な交差判定を解かなくても、SDFさえ定義すれば任意の形状に対して近似的にレイと表面の交点が求まるのがこの手法の特徴です。

### パストレーシングとコサイン重み付き半球サンプリング

拡散反射（艶消しのマットな表面）は、光が当たった点から法線を軸にした半球状のあらゆる方向へランダムに反射します。パストレーシングでは、この挙動を乱数でランダムな方向を1つ抽選してレイを継続することでシミュレートします（`cosineSampleHemisphere()`）。「コサイン重み付き」とは、法線に近い方向ほど選ばれる確率を高くするサンプリング方法です。

レンダリング方程式をモンテカルロ法で推定すると次の形になります。

```
L_o ≈ (1/N) Σ [ f_r · L_i · cosθ / pdf(ω) ]
```

ここで`cosθ`は法線とサンプリング方向の内積です。Lambertian BRDF `f_r = albedo / π`と、コサイン重み付きサンプリングのPDF `pdf(ω) = cosθ / π`を代入すると、

```
f_r · cosθ / pdf(ω) = (albedo/π) · cosθ / (cosθ/π) = albedo
```

`cosθ`が分子・分母で相殺されます。そのため実装上は`throughput *= albedo`という単純な掛け算だけで済んでおり、`dot(法線, サンプリング方向)`を別途明示的に計算する必要がありません。内積（cosθ）が消えたわけではなく、「法線方向に近いほど選ばれやすい」というサンプリングの偏りとしてコードに暗黙的に埋め込まれている、という点がポイントです（もし全方向一様サンプリングにした場合は`pdf = 1/(2π)`固定になるため、`albedo * dot(法線, 方向) * 2.0`のように明示的にcosθを掛ける必要があります）。

### 時間方向の積分（累積平均）によるノイズ収束

カメラは時間で動かさず固定したまま、フレームごとに新しいノイズ入りのパストレーシング結果を、前フレームまでの累積平均に混ぜ込んでいきます（ping-pongさせた2枚の`rgba16float`テクスチャで実装）。

```
newAvg = prevAvg + (currentSample - prevAvg) / n   // n = 累積フレーム数
```

これは総和を保持せずに逐次的に平均を更新する定番の手法で、フレームを重ねるほどモンテカルロサンプリングによるノイズが打ち消されて画像が滑らかに収束していきます。画面右上のスライダーで「反射回数上限」「レイ数/pixel」を変更すると、蓄積フレーム数が0にリセットされ、新しい設定で収束し直します。

## 開発中に見つかった落とし穴

### 夕焼けの色が付かなかった問題(`shaders/analytic-sky.html`)

大気散乱シェーダーの最初の実装では、視線方向の光路長(水平線に近いほど大気を長く通る)だけを減衰の要因にしていた。式で書くと、

```
viewExtinction = exp(-viewOpticalDepth * (rayleighCoeff + mieCoeff) * scale)
inscatter = (rayleighCoeff*rayleighPhase + mieCoeff*miePhase) * viewExtinction
```

これだけだと、離れた方角(太陽から離れた斜め上とか)を見てもほぼ同じような灰色っぽい色にしかならず、正午の青空も夕焼けのオレンジも弱々しくにじんだような色にしかならなかった。

原因は、**太陽光そのものが地表に届くまでに大気を通ってきた分の減衰**をモデルに入れていなかったこと。夕焼けが赤くなるのは、太陽が低い角度にあるときは太陽光が長い距離大気中を進んでくる間に青い波長がRayleigh散乱で強く散乱されて抜け落ち、赤系の波長だけが生き残って届くから。視線方向の減衰だけでは「見ている方向によって色が変わる」効果しか再現できず、「太陽の高さによって空全体の色調が変わる」効果が抜けていた。

修正として、太陽の仰角(`sunDir.y`)から求めた光路長でも同様に波長依存の減衰(`sunTransmittance`)を計算し、視線方向の減衰と掛け合わせるようにした。

```
sunTransmittance = exp(-sunOpticalDepth * (rayleighCoeff + mieCoeff) * scale)
inscatter = (rayleighCoeff*rayleighPhase + mieCoeff*miePhase) * viewExtinction * sunTransmittance
```

これで太陽が水平線に近づくと`sunTransmittance`の青チャンネルだけが大きく減衰し、赤チャンネルが相対的に生き残ることで、ちゃんとオレンジ色の夕焼けになった。太陽本体(円盤)の色も`sunTransmittance`で染めるようにしたので、正午は白っぽく、夕方は赤っぽく見えるようになっている。

オフスクリーンでピクセル値を読み出して検証していたが、修正前は「太陽から離れた斜め方向」と「太陽方向(円盤を避けた位置)」のRGB比がほとんど同じ(せいぜい数%の差)だったのに対し、修正後は明確にR>G>Bとなり(例: 夕焼けで`[116, 86, 44]`)、数値上でも違いが確認できた。

### WGSLの予約語との衝突

カメラの注視点を表す変数名として`target`を使ったところ、`error: 'target' is a reserved keyword`でシェーダーのコンパイルに失敗した。WGSLにはこの手の予約語がいくつかあり、うっかり変数名にすると分かりにくいエラーにはならず一発でコンパイルエラーになるので気づきやすいが、変数名の選択には注意が必要(このプロジェクトでは`lookAt`のような別名に置き換えて回避した)。

### ヘッドレス環境でのcanvas読み出しの制限

このセッションのようなGPUパススルーの無いサンドボックス環境(swiftshaderによるソフトウェアレンダリング)では、WebGPUの初期化やオフスクリーンテクスチャへのレンダリング自体は問題なく動くが、実際に画面に表示される`<canvas>`の内容を`drawImage`やスクリーンショットで読み出すと常に透明/黒になった。そのため、動作確認は「画面のcanvasを直接読み出す」のではなく、「同じシェーダーロジックを使ってオフスクリーンのテクスチャに描画し、`copyTextureToBuffer`で直接ピクセル値を読み出す」という方法に切り替えて行っている。実機のブラウザで見た目を確認する場合はこの制限は無関係。

### パラメータチューニングを、実際のカメラ/太陽の向きと無関係なテスト値でやってしまった

空の色(`shaders/analytic-sky.html`)を「もっと空色っぽく」調整したとき、最初は自分で適当に選んだ視線方向・太陽方向のベクトル(例:「太陽から少し離れた斜め上」)を使って`skyColor()`だけを単体テストし、その出力が目標の色(`[135, 206, 235]`)に近づくようRayleigh/Mie係数をNode.js上で数値的に最適化した。単体テストとしては目標色にほぼ一致する結果が出たが、実際のシェーダー(固定カメラの向き+太陽が時間で動く軌道の式)に同じ係数を適用してオフスクリーンで検証し直すと、画面中央付近が真っ白に近い色になってしまい、テストで見ていた色とはまったく違う結果になった。

原因は、テストに使った視線方向・太陽方向のベクトルが、実際のカメラの向き(`forward = normalize(vec3(0, 0.35, -1))`)や太陽の軌道式(`sunDir = normalize(vec3(cos(sunAngle)*0.35, sin(sunAngle), -0.9))`)とかけ離れた、自分で適当に決めた値だったこと。特に「正午」のテストでは太陽の仰角を`sunDir.y ≈ 1`(真上)と仮定していたが、実際の軌道式では太陽が単純な円運動ではなく`z`成分が固定なので、正規化後の最大仰角は`0.74`程度にしかならない。テスト用のベクトルと実際にレンダリングされる視線・太陽の位置関係(特に`dot(視線, 太陽方向)`の値)がずれていたため、そのテストに対して最適化したパラメータは実際の画面には通用しなかった。

対策として、実際のカメラの基底ベクトル(`forward`/`right`/`up`)と太陽の軌道式をNode.js側にもそのまま複製し、実際に画面に表示される方向(画面中央など)でパラメータを検証し直した。**シェーダーの一部分だけを単体テストする場合、その入力(視線方向や太陽方向など)が実際のレンダリングで発生する値の範囲を代表しているか確認しないと、テストで良い結果が出てもそれは実物とは無関係な結果になりうる**、という教訓。
